# 趋势驱动新品机会系统：产品边界与分层架构

状态：已确认的产品基线  
日期：2026-07-17  
适用范围：后续产品设计、数据建模、算法选择、页面改造和外部能力接入

## 1. 产品初心

从全球新闻趋势中，尽早发现可能引发新消费场景、新需求或新约束的变化，并把它整理成值得进一步验证的新品机会线索。

系统的核心问题不是“今天什么新闻最热”，也不是“自动生成多少商品”，而是：

> 这个正在发生的变化，是否可能孕育一个尚未被充分满足的新品机会？

## 2. 第一阶段产品定位

第一阶段聚焦于适合电商销售的实体消费品新品机会。

### 2.1 目标用户

- 寻找新品方向的独立卖家、产品经理和小型选品团队。
- 希望更早发现需求变化，但无法持续人工跟踪全球信息的研究人员。
- 需要用证据筛选机会，而不是依赖爆款榜单或主观灵感的决策者。

### 2.2 用户核心任务

每天用有限时间完成三件事：

1. 看见真正发生变化的全球趋势。
2. 判断哪些变化可能产生新的实体商品需求。
3. 选择少量机会进入具体商品构思和市场验证。

### 2.3 第一阶段不包含

- 软件、订阅、咨询、课程、资料包和纯内容产品。
- 人物、影视、赛事等未经授权的事件周边。
- 依赖医疗功效、危险品或明显高风险合规声明的商品。
- 无法在合理打样、生产和运输周期内落地的纯短时热点。
- 没有平台证据时直接给出“最终选品推荐”。
- 用主题词、新闻关键词或模型自信度冒充真实购买需求。

## 3. 产品对象必须分层

系统不得再混用“趋势”“机会”“商品假设”和“推荐”。它们是证据强度不同的四种产品对象。

```mermaid
flowchart LR
    A["TrendEvent\n趋势事件"] --> B["OpportunitySignal\n新品机会线索"]
    B --> C["ProductHypothesis\n具体商品假设"]
    C --> D["MarketEvidence\n平台市场证据"]
    D --> E["ValidatedRecommendation\n已验证选品推荐"]
```

### 3.1 TrendEvent：趋势事件

事实层，只描述发生了什么和变化有多强。

必须包含：

- 原始来源和链接。
- 市场、语言和信号类型。
- 首次出现、最近出现和持续时间。
- 排名、速度、跨来源覆盖和趋势分。
- 聚类成员及合并依据。

不得包含：商品推荐、购买需求结论和市场销量推断。

### 3.2 OpportunitySignal：新品机会线索

判断层，描述为什么一个趋势可能影响消费，以及它是否值得进入下一步。

建议字段：

- `event_id`
- `change_type`：行为变化、环境约束、法规变化、技术能力、人口/生活方式变化等。
- `consumer_relevance_score`
- `product_opportunity_score`
- `target_users`
- `new_scenarios`
- `unmet_needs`
- `related_product_categories`
- `durability`：短时、季节性、中期、结构性。
- `lead_time_fit`：是否来得及开发和供应。
- `evidence_ids`
- `confidence`
- `missing_evidence`
- `review_status`：待看、值得跟进、无商品机会、需补证据。

这一层可以指出相关商品空间，但不负责创造具体商品名称。

### 3.3 ProductHypothesis：具体商品假设

构思层，描述可能做什么新品。它可以由人工、大模型或外部研究 Agent 产生，但必须引用一条 OpportunitySignal。

必须包含：

- 对应机会线索。
- 具体实体商品形态。
- 目标人群和使用场景。
- 要解决的新问题或新约束。
- 与现有商品相比的预期差异。
- 可验证的商品中心词和查询词候选。
- 生成方式、模型/人员、版本和引用证据。
- 状态：草稿、待市场验证、已否决、已验证。

这仍然是待验证假设，不得直接进入最终推荐榜。

### 3.4 MarketEvidence：平台市场证据

验证层，由 Amazon 等平台数据、人工成本数据或其他可靠数据源提供。

可以包含：

- 搜索需求和变化。
- 购买意图代理。
- 竞品数量和头部集中度。
- 售价和价格分布。
- 评论痛点和未满足需求。
- 退货率。
- 成本、平台费、广告和贡献毛利。
- 数据来源、采集时间和原始文件哈希。

### 3.5 ValidatedRecommendation：已验证选品推荐

决策层。只有 ProductHypothesis 获得足够 MarketEvidence，并通过风险与经济性门槛后才能生成。

最终推荐排序只能使用已经定义、可追溯的趋势证据和市场证据。缺少市场数据时保持“待验证”，不得用主观假设分代替市场分。

## 4. 当前程序边界

当前无生成模型、无直接平台数据接入时，程序只负责：

1. 全球趋势数据采集。
2. 去重、聚类和事件合并。
3. 趋势强度和变化速度计算。
4. 消费相关性与新品机会相关性的机器学习判断。
5. 生成 OpportunitySignal 机会简报。
6. 收集人工反馈，形成后续训练数据。

当前程序不负责：

- 生成看似具体但没有依据的产品名称。
- 自动生成可信的 Amazon 查询词。
- 自动填写购买意图、差异化和可行性分数。
- 在缺少平台数据时给出最终机会分。
- 把未验证商品假设推入推荐榜或外部通知。

## 5. 机器学习与深度学习边界

非生成式模型的核心职责是“发现、分类和排序”，不是“创造商品”。

### 5.1 可直接应用的能力

| 任务 | 初始方法 | 数据成熟后的方法 |
|---|---|---|
| 语义去重和事件聚类 | 多语言文本向量 + 密度/层次聚类 | 基于人工合并反馈优化阈值或度量学习 |
| 趋势异常发现 | 移动基线、变化率、变化点检测 | 针对来源和类目的时间序列模型 |
| 消费相关性 | 关键词弱标签 + 向量相似度 | 监督分类器 |
| 新品机会相关性 | 人工审核优先 | 文本向量和趋势特征上的 Logistic Regression/LightGBM，再逐步微调编码模型 |
| 变化类型识别 | 多标签弱规则 | 多标签分类模型 |
| 商品类目关联 | 类目原型向量检索 | 基于人工修正训练类目分类器 |
| 机会排序 | 趋势分 + 可解释特征 | Learning-to-Rank 或校准后的分类概率 |

### 5.2 训练数据闭环

产品必须提供低成本反馈入口：

- 值得跟进。
- 没有实体商品机会。
- 消费关联太弱。
- 只是短时热点，来不及落地。
- 商品类目判断错误。
- 证据不足，暂缓。

这些反馈与当时的新闻文本、趋势特征、证据和模型版本一起保存。早期使用弱规则和人工审核建立基线，积累数百到数千条高质量样本后，再决定是否微调深度模型。

### 5.3 模型输出限制

- 分类概率不得直接显示为“市场需求”。
- 模型置信度只表示模型判断稳定性，不代表商业价值。
- 任何 OpportunitySignal 都必须能够回到原始证据。
- 模型必须允许弃权，不能为了填满榜单而输出机会。

## 6. 后续生成模型边界

未来的大模型只负责从 OpportunitySignal 生成 ProductHypothesis，不替代系统事实、状态和评分。

建议接口：

```python
class ProductHypothesisGenerator(Protocol):
    async def generate(
        self,
        signal: OpportunitySignal,
        evidence: list[Evidence],
    ) -> ProductHypothesisResult: ...
```

可插拔实现：

- `HumanHypothesisGenerator`
- `LocalModelHypothesisGenerator`
- `CloudModelHypothesisGenerator`
- `ExternalAgentHypothesisGenerator`

生成结果必须经过统一 Schema、实体商品规则、风险规则和证据引用检查。更换模型不得改变下游数据结构。

## 7. 平台数据与 Agent/Skill 边界

### 7.1 MarketplaceDataProvider

Amazon、Walmart 或其他平台通过统一 Provider 写入 MarketEvidence：

```python
class MarketplaceDataProvider(Protocol):
    async def validate(
        self,
        hypothesis: ProductHypothesis,
    ) -> MarketEvidenceResult: ...
```

Provider 可以是平台 API、Seller Central CSV、授权数据源或人工录入。它只提供证据，不决定最终推荐。

### 7.2 Agent 和 Skill

Agent/Skill 是高级研究与人工协作层，不是核心定时管道。

它们可以：

- 读取机会线索和证据。
- 生成或修正商品假设。
- 调用多个平台工具补充数据。
- 形成深度研究报告。
- 经人工确认后写回系统。

它们不能：

- 绕过数据状态和权限直接发布推荐。
- 替代系统数据库成为唯一事实来源。
- 绕过市场证据和风险门槛。
- 用模型自然语言结论直接覆盖确定性评分。

## 8. 产品信息架构

### 8.1 当前阶段

1. **全球趋势**：展示趋势事件、变化速度和来源证据。
2. **新品机会线索**：展示值得人工关注的 OpportunitySignal。
3. **反馈队列**：快速标记值得跟进、无机会或判断错误。

### 8.2 后续阶段

4. **商品假设**：具体产品构思及其证据和生成来源。
5. **市场验证**：平台数据、缺失项和验证进度。
6. **已验证推荐**：只有通过市场和风险门槛的商品。

不同阶段必须使用不同名称、状态和视觉层级，避免用户把机会线索误认为推荐结论。

## 9. 现有功能处置清单

| 现有能力 | 处置 | 说明 |
|---|---|---|
| 全球数据源采集 | 保留 | 属于核心事实层 |

| 新闻聚类和趋势评分 | 保留并升级 | 后续引入语义向量和时间序列方法 |

## 10. 核心产品指标

### 当前阶段

- `Opportunity Precision@K`：Top K 线索中人工认为值得跟进的比例。
- 无消费意义事件占比。
- 重复事件占比。
- 商品类目判断准确率。
- 从趋势首次出现到机会线索生成的时间。
- 人工审核一条线索的平均时间。
- 模型弃权率与有效机会率。

### 后续阶段

- 机会线索转化为商品假设的比例。
- 商品假设通过市场验证的比例。
- 样品测试率、上架率和淘汰原因。
- 7 天、30 天和更长期结果。
- 不同模型、提示词和数据源版本的验证通过率。

新闻数量、生成商品数量和榜单是否填满不是成功指标。

## 11. 分阶段实施计划

### Phase 0：产品重置

- 确认实体消费品边界。
- 停止把固定模板产物称为选品推荐。
- 停止用假设分代理市场分。
- 建立本文档作为后续变更约束。

验收：任何未完成平台验证的对象都不会显示为“已验证推荐”。

### Phase 1：OpportunitySignal

- 新增机会线索数据结构和状态。
- 将趋势事件与具体商品假设拆开。
- 建立消费相关性、新品机会相关性、变化类型和类目字段。
- 增加快速人工反馈。

验收：系统可以只输出趋势或机会线索，并允许零机会。

### Phase 2：机器学习基线

- 引入多语言文本向量用于聚类和类目检索。
- 用人工反馈训练第一版可解释分类/排序模型。
- 建立离线评测集和版本对比。

验收：以人工评审的 Precision@K 为主要指标，模型效果优于规则基线后才上线。

### Phase 3：商品假设生成

- 首先支持人工创建商品假设。
- 接入可插拔生成模型 Provider。
- 建立结构校验、证据引用和实体商品约束。

验收：更换人工、本地模型或云模型，不影响下游数据结构和验证流程。

### Phase 4：平台验证与推荐

- 将现有 Seller Central CSV 能力封装为 MarketplaceDataProvider。
- 增加其他平台或授权数据源。
- 只对完成验证的商品计算最终推荐排序。

验收：每条推荐可以回溯到趋势事件、机会线索、商品假设和平台证据。

## 12. 产品决策原则

后续设计和开发必须遵守：

1. 事实、判断、假设和验证结果分层保存。
2. 没有证据时允许为空，不强行填榜。
3. 模型用于提高发现效率，不替代市场证据。
4. 越接近最终推荐，证据门槛越高。
5. 系统输出名称必须匹配真实证据强度。
6. 先建立反馈闭环，再追求复杂模型。
7. 所有模型、规则、提示词和数据源均可版本化、替换和回溯。
