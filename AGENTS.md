# 文档与开发治理

本文件只规定仓库文档职责、权威顺序和开发纪律，不定义产品流程或技术实现细节。

## 文档职责

- `README.md` 是项目入口，只提供项目摘要、真实流程概览、启动方式和文档导航。
- `docs/workflow-contract.md` 是业务流程、节点合同、停止条件和 AI 边界的唯一真相来源。
- `docs/architecture.md` 是模块关系、技术结构、数据对象关系、兼容结构和冻结能力的唯一真相来源。
- `HANDOFF.md` 只记录当前开发交接、已知缺口、本轮工作和下一项任务；不得长期承载产品规格。
- `skills/` 中的 Markdown 是运行时研究规则，必须服从业务流程合同和可执行代码的安全约束。
- `docs/archive/` 中的 Markdown 只供历史参考，不是当前实现、流程或架构依据。
- 专项操作文档只描述各自操作范围，不重新定义完整流程。

## 权威顺序

1. 当前可执行代码和数据库 Schema：当前实现事实。
2. `docs/workflow-contract.md`：业务流程和 AI 边界。
3. `docs/architecture.md`：技术结构。
4. `README.md`：项目摘要。
5. `HANDOFF.md`：临时开发状态。
6. `skills/`：研究执行规则。
7. `docs/archive/`：历史参考。

发现文档之间或文档与代码冲突时，不得静默选择或继续复制冲突内容；必须明确指出冲突，并按职责修正对应权威文档。

## 变更纪律

- 修改业务流程、节点条件、停止条件或 AI 边界时，必须同步更新 `docs/workflow-contract.md`。
- 修改模块关系、状态存储、API 分层、兼容结构或冻结能力时，必须同步更新 `docs/architecture.md`。
- 不得把计划能力、接口占位或测试替身描述为已实现、已接入或已验收。
- 不得在多个文档复制完整节点合同、Schema、架构说明或长篇状态记录；应链接到权威文档。
- 没有明确、长期且不重叠的职责时，不得新建 Markdown。
- 小功能不得顺便进行大重构；变更范围必须与任务目标一致。
- 历史设计需要保留时移入 `docs/archive/`，并明确标记为非权威。
- 临时数据库数量、测试数量、分支工作树和外部服务结果不得写成长期产品事实。
- 修改完成后应检查内部链接、文档职责边界，以及“当前实现”和“目标行为”是否明确分开。
